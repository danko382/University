\textbf{Жизненный цикл:} \smallskip
\begin{enumerate}
    \item Организационные проекты
    \begin{enumerate}
        \item Управление проектом
        \item Создание инфраструктуры
        \item Оценка и улучшение жизненного цикла
        \item Обучение
    \end{enumerate}
    \item Основные процессы
    \begin{enumerate}
        \item Приобретение
        \item Поставка
        \item Разработка
        \item Эксплуатация
        \item Сопровождение
    \end{enumerate}    
    \item Вспомогательные процессы
    \begin{enumerate}
        \item Документирование
        \item Управление конфигурацией
        \item Обеспечение качества
        \item Верификация
        \item Аттестация
        \item Совместная оценка
        \item Аудит
        \item Разрешение проблем
    \end{enumerate}
\end{enumerate}  

\textbf{В компаниях, плохо работающих с ресурсами,}
лучше работать только в целях набора опыта. \smallskip

\textbf{Сотрудника} могут принять либо в штат, либо на проект (пока идёт
проект). Второй вариант используют компании, как правило, плохо
управляющие проектом. \smallskip

\textbf{В жизненном цикле важно} следить за совместимостью ПО с 
железом. \smallskip

\textbf{Хорошее внедрение} лучше хорошего разработчика. Внедрение 
очень важно! \smallskip

\textbf{70 лет = очень старый.} \smallskip

\textbf{Если вы работаете} с облаками важно иметь разрешение 
на эксплуатацию в нём (в договоре). \smallskip

\textbf{Верификация,} т.е. в принципе программа работает
верно. \smallskip

\textbf{Обеспечение качества} = проверка по ТЗ. \smallskip

\textbf{На аудит} могут приглашатся сторонние оценщики. \smallskip

\textbf{ПО в компаниях с жёсткими} требованиями к безопасности
не должно иметь незадокументированных дополнительных
возможностей. \smallskip

\textbf{Заказчиком принимаются следующие действия:} \smallskip

Осознание своих потребностей в программной системе и принятие решения
относительно покупки, разработки под заказ или усовершенствования
существующей системы. \smallskip

Подготовка заявочных предложений, содержащих требования к системе,
условия её функционирования и технические ограничения, а также 
другие условия контракта. \smallskip

\textbf{Процесс разработки включает в себя:} \smallskip

Оформление проектной и эксплуатационной документации. \smallskip

Подготовку материалов, необходимых для проверки работоспособности
программного продукта и его соответствия стандартам
качества. \smallskip

Разработку материалов необходимых для подготовки и обучения
персонала. \smallskip

Выбор модели жизненного цикла. \smallskip

Анализ требований к системе. \smallskip

Проектирование архитектуры системы. \smallskip

Анализ программных требований. \smallskip

Детальное конструирование ПО. \smallskip

Конструирование и тестирование ПО. \smallskip

Интеграция ПО. \smallskip

Квалификационное тестирование ПО. \smallskip

Интеграция системы. \smallskip

Квалификациооное тестирование системы. \smallskip

Установка ПО. \smallskip

Приёмка ПО. \smallskip

\textbf{Анализ требований к системе:} \smallskip

На данном этапе рассматривается область применения системы. Список
требований к разрабатываемой системе должен включать. \bigskip

Совокупность условий, при которых предполагается эксплуатировать
будущую систему. \smallskip

Описание выполняемых системой функций. \smallskip

Ограничения в процессе разработки. \smallskip

\textbf{Анализ требований к ПО:} \smallskip

Предподагается определение следующих характеристик для
каждого компонента ПО: \smallskip

Функциональных фозможностей, включая характеристики производительности 
и среды функционирования компонента. \smallskip

Внешних интерфейсов. \smallskip

Спецификация надёёжности и безопасности. \smallskip

Эргономических требований. \smallskip

Требований к используемым данным. \smallskip

Требований к установке и приёмке. \smallskip

Требований к пользовательской документации. \smallskip

Требований к эксплуатации и сопровождению. \smallskip

\textbf{функиональные возможности говорит заказчик}, характеристики
производительности тоже, желательно, сообщает заказчик. \smallskip

\textbf{Если что"=то стирается из БД, то лучше потребовать нормативы}
по хранению данных организации. \smallskip

\textbf{Наработка на отказ}~--- время работы без перерыва, идеал
99,9 процентов часов работы за год. \smallskip

\textbf{Эргономические требования =} вопрос об интерфейсах. \smallskip

\textbf{Презентация строго чёрными} буквами на белом фоне, если 
у Батраевой попадётесь, то "затопчит и уничтожит". \smallskip

\textbf{Интерфейсы должны быть стандартизированными.} \smallskip

\textbf{Докуметнация} по ПО должна быть исчерпывающим
для пользователя. \smallskip

\smallskip{У SQL сервера microsoft} лучшая, по мнению Батраевой,
документация. \smallskip

\textbf{В рамках проектирования архитектуры для каждого
компонента ПО выполняются следующие задчи:} \smallskip

Определение на высоком уровне абстракции структуры программного
обеспечения и состава его компонентов. \smallskip

Разработка и документирование программных интерфейсов ПО и
баз данных. \smallskip

Разработка предварительной версии пользовательской
документации. \smallskip

Разработка и документирование предварительных требований
к тестам и планам интеграции ПО. \smallskip

\textbf{Детальное конструирование ПО включает в себя:} \smallskip

Описание компонетов ПО и интерфейсов между ними в объёме 
достаточном для их последующего самостоятельного кодирования
и тестирования. \smallskip

Разработку и документирование детального проекта базы данных. \smallskip

Обновление пользовательской документации. \smallskip

Разработку и документирование требований к тестам и плана 
тестирования компонентов. \smallskip

Обновление плана интеграции ПО. \smallskip

\textbf{Квалификационное тестирование ПО} проводится разработчиком
в присутствии заказчика для демонстрации того, что ПО удовлетворяет
своим спецификациям и готово к использованию в условиях
эксплуатации. \smallskip

При этом также проверяется полнота технической и пользовательской
документации и её адекватность компонентам ПО. \smallskip

\textbf{Установка ПО} осуществляется разработчиком в соответсвии
с планом в той среде и на том оборудовании, которые предусмотрены
договором. В процессе установки проверяется работоспособность ПО и
баз данных. \smallskip

\textbf{Приёмка ПО} предусматривает оценку результата
квалификационного тестирования системы и документрирование результатов
оценки, которое производится заказчиком с помощью разработчика.
Разработчик выполняет окончательную передачу ПО заказчику в соответсвии
с договором, обеспечивая при этом необходимое обучние и
поддержку. \smallskip

\textbf{Функции службы сопровождения:} \smallskip

Анализ проблем и запросов на модификацию ПО. \smallskip

Модификация программного продукта. \smallskip

Его проверка и приёмка. \smallskip

Перенос ПО в другую среду. \smallskip

Снятие ПО с эксплуатации. \smallskip

\textbf{Снятие ПО с эксплуатации} = архивизация данных в читабельный
формат + стирание программы. \smallskip
